Lazy原生类型

最近 Add lazy initialization primitives to std #72414 合并进标准库，以前为了实现全局变量的初始化， 一般有2种方式：

  1. const fn
  2. 第三方crate，比如once_cell 和 lazy_static

由于const fn同常量表达式一样，是在const context上下文中执行，即在编译时计算函数（或表达式）结果，因此在使用上有诸多限制，比如类型参数只能绑定 生命周期和Sized（?Sized），只能调用其他const fn等。

Lazy类型实现的是延迟初始化，因此不必受到const fn的限制，同时能加快某些组件的启动，未使用到的功能甚至可以不初始化。

延迟初始化应用广泛，比如配置文件的读取，异步运行时需要初始化全局的Executor等，因此合并进标准库可以方便不少。

在std::lazy下有2个用于延迟初始化的类型：Lazy和SyncLazy。其中Lazy是非线程安全的，主要用于TLS或非多线程访问。SyncLazy则是Lazy的线程安全 版本，多用于全局变量的初始化。

Lazy类型相对简单，因此这篇文章主要介绍SyncLazy的实现：

#[unstable(feature = "once_cell", issue = "74465")]
pub struct SyncLazy<T, F = fn() -> T> {
    cell: SyncOnceCell<T>,
    init: Cell<Option<F>>,
}

#[unstable(feature = "once_cell", issue = "74465")]
unsafe impl<T, F: Send> Sync for SyncLazy<T, F> where SyncOnceCell<T>: Sync {}

cell用于保存内部数据，init则用于保存初始化函数，当第一次访问数据时，init中保存的函数将会执行。

由于SyncLazy需要支持多线程的共享访问，所以需要实现Sync。

#[unstable(feature = "once_cell", issue = "74465")]
pub struct SyncOnceCell<T> {
    once: Once,
    // Whether or not the value is initialized is tracked by `state_and_queue`.
    value: UnsafeCell<MaybeUninit<T>>,
}

在SyncOnceCell内，value用于保存数据 ，once用于确保在多线程环境下，初始化函数只被执行一次，这也是和非线程安全版Lazy的最大区别：

#[unstable(feature = "once_cell", issue = "74465")]
pub struct OnceCell<T> {
    // Invariant: written to at most once.
    inner: UnsafeCell<Option<T>>,
}

通过实现Deref，可以像智能指针那样使用内部类型：

impl<T, F: FnOnce() -> T> SyncLazy<T, F> {
    #[unstable(feature = "once_cell", issue = "74465")]
    pub fn force(this: &SyncLazy<T, F>) -> &T {
        this.cell.get_or_init(|| match this.init.take() {
            Some(f) => f(),
            None => panic!("Lazy instance has previously been poisoned"),
        })
    }
}

#[unstable(feature = "once_cell", issue = "74465")]
impl<T, F: FnOnce() -> T> Deref for SyncLazy<T, F> {
    type Target = T;
    fn deref(&self) -> &T {
        SyncLazy::force(self)
    }
}

在deref方法内调用了force方法，因此关键的部分就是SyncOnceCell的get_or_init方法实现了：

#[unstable(feature = "once_cell", issue = "74465")]
pub fn get_or_init<F>(&self, f: F) -> &T
where
	F: FnOnce() -> T,
{
	match self.get_or_try_init(|| Ok::<T, !>(f())) {
		Ok(val) => val,
	}
}

#[unstable(feature = "once_cell", issue = "74465")]
pub fn get_or_try_init<F, E>(&self, f: F) -> Result<&T, E>
where
	F: FnOnce() -> Result<T, E>,
{
	// Fast path check
	// NOTE: We need to perform an acquire on the state in this method
	// in order to correctly synchronize `SyncLazy::force`. This is
	// currently done by calling `self.get()`, which in turn calls
	// `self.is_initialized()`, which in turn performs the acquire.
	if let Some(value) = self.get() {
		return Ok(value);
	}
	self.initialize(f)?;

	debug_assert!(self.is_initialized());

	// Safety: The inner value has been initialized
	Ok(unsafe { self.get_unchecked() })
}

到这里还是比较常规的操作，先看数据是否已经初始化，如果是就直接返回，否则执行初始化。

#[cold]
fn initialize<F, E>(&self, f: F) -> Result<(), E>
where
	F: FnOnce() -> Result<T, E>,
{
	let mut res: Result<(), E> = Ok(());
	let slot = &self.value;

	// Ignore poisoning from other threads
	// If another thread panics, then we'll be able to run our closure
	self.once.call_once_force(|p| {
		match f() {
			Ok(value) => {
				unsafe { (&mut *slot.get()).write(value) };
			}
			Err(e) => {
				res = Err(e);

				// Treat the underlying `Once` as poisoned since we
				// failed to initialize our value. Calls
				p.poison();
			}
		}
	});
	res
}

initialize方法内会执行once的call_once_force方法，闭包的参数p类型是OnceState，这个类型描述了初始化的状态，如果闭包f执行失败，则会调用poison 方法将状态置为POISONED，代表数据没有完全初始化完成。

NOTE: 不过从get_or_init方法中可以看到，闭包f的Err类型为never（!），因此永远不会失败。但是OnceState的状态在初始化线程panic后也会置为 POISONED，可以从后面的代码里看到。

NOTE: #[cold]这个attr告诉编译器这个方法会很少调用（因为一旦初始化完成，后面就不会再执行了），这个参数会传递给llvm用于代码优化。

#[unstable(feature = "once_poison", issue = "33577")]
pub fn call_once_force<F>(&self, f: F)
where
	F: FnOnce(&OnceState),
{
	// Fast path check
	if self.is_completed() {
		return;
	}

	let mut f = Some(f);
	self.call_inner(true, &mut |p| f.take().unwrap()(p));
}

#[cold]
fn call_inner(&self, ignore_poisoning: bool, init: &mut dyn FnMut(&OnceState)) {
	let mut state_and_queue = self.state_and_queue.load(Ordering::Acquire);
	loop {
		match state_and_queue {
			COMPLETE => break,
			POISONED if !ignore_poisoning => {
				// Panic to propagate the poison.
				panic!("Once instance has previously been poisoned");
			}
			POISONED | INCOMPLETE => {
				// Try to register this thread as the one RUNNING.
				let old = self.state_and_queue.compare_and_swap(
					state_and_queue,
					RUNNING,
					Ordering::Acquire,
				);
				if old != state_and_queue {
					state_and_queue = old;
					continue;
				}
				// `waiter_queue` will manage other waiting threads, and
				// wake them up on drop.
				let mut waiter_queue = WaiterQueue {
					state_and_queue: &self.state_and_queue,
					set_state_on_drop_to: POISONED,
				};
				// Run the initialization function, letting it know if we're
				// poisoned or not.
				let init_state = OnceState {
					poisoned: state_and_queue == POISONED,
					set_state_on_drop_to: Cell::new(COMPLETE),
				};
				init(&init_state);
				waiter_queue.set_state_on_drop_to = init_state.set_state_on_drop_to.get();
				break;
			}
			_ => {
				// All other values must be RUNNING with possibly a
				// pointer to the waiter queue in the more significant bits.
				assert!(state_and_queue & STATE_MASK == RUNNING);
				wait(&self.state_and_queue, state_and_queue);
				state_and_queue = self.state_and_queue.load(Ordering::Acquire);
			}
		}
	}
}

call_once_force方法又会调用call_inner，而这个call_inner就是once最核心的方法了。

这个方法首先会先获取state_and_queue属性，这个类型本身看定义的话是个AtomicUsize：

#[stable(feature = "rust1", since = "1.0.0")]
pub struct Once {
    // `state_and_queue` is actually an a pointer to a `Waiter` with extra state
    // bits, so we add the `PhantomData` appropriately.
    state_and_queue: AtomicUsize,
    _marker: marker::PhantomData<*const Waiter>,
}

但其实state_and_queue是一个Waiter链表指针和初始化状态的复合，其中低2bit为状态位。之所以能这么实现，是因为Waiter类型的对齐属性：

#[repr(align(4))] // Ensure the two lower bits are free to use as state bits.
struct Waiter {
    thread: Cell<Option<Thread>>,
    signaled: AtomicBool,
    next: *const Waiter,
}

#[repr(align(4))]的对齐属性保证了Waiter指针的低2bit一定是0。

在call_inner中最重要的2个分支是POISONED | INCOMPLETE和_，其中POISONED | INCOMPLETE分支说明初始化还未执行过或者执行了初始化函数的线程panic了， 导致数据依然是不可用的，因此这个分支需要执行初始化函数。而_分支说明另一个线程正在执行初始化函数的过程中，因此这个分支需要将当前线程注册到 等待队列里，等待被唤醒。

fn wait(state_and_queue: &AtomicUsize, mut current_state: usize) {
    // Note: the following code was carefully written to avoid creating a
    // mutable reference to `node` that gets aliased.
    loop {
        // Don't queue this thread if the status is no longer running,
        // otherwise we will not be woken up.
        if current_state & STATE_MASK != RUNNING {
            return;
        }

        // Create the node for our current thread.
        let node = Waiter {
            thread: Cell::new(Some(thread::current())),
            signaled: AtomicBool::new(false),
            next: (current_state & !STATE_MASK) as *const Waiter,
        };
        let me = &node as *const Waiter as usize;

        // Try to slide in the node at the head of the linked list, making sure
        // that another thread didn't just replace the head of the linked list.
        let old = state_and_queue.compare_and_swap(current_state, me | RUNNING, Ordering::Release);
        if old != current_state {
            current_state = old;
            continue;
        }

        // We have enqueued ourselves, now lets wait.
        // It is important not to return before being signaled, otherwise we
        // would drop our `Waiter` node and leave a hole in the linked list
        // (and a dangling reference). Guard against spurious wakeups by
        // reparking ourselves until we are signaled.
        while !node.signaled.load(Ordering::Acquire) {
            // If the managing thread happens to signal and unpark us before we
            // can park ourselves, the result could be this thread never gets
            // unparked. Luckily `park` comes with the guarantee that if it got
            // an `unpark` just before on an unparked thread is does not park.
            thread::park();
        }
        break;
    }
}

把当前线程添加到等待队列的实现方式是把当前Waiter添加到链表表头。

而执行初始化的线程首先需要将初始化状态置为RUNNING，随后创建WaiterQueue对象，WaiterQueue对象上保存了等待队列，同时初始化set_state_on_drop_to 值为POISONED，这个属性是指当WaiterQueue对象析构后，once的状态需要置为对应的值，因此如果发生了panic，则once的状态就是 POISONED。当初始化函数执行完，则set_state_on_drop_to会设置为init_state中的set_state_on_drop_to。而init_state会作为参数传递到初始化函数中， 也就是说初始化函数中可以修改set_state_on_drop_to。

NOTE: 上面所说的初始化函数是指被包装过的，而不是用户的初始化函数。

impl Drop for WaiterQueue<'_> {
    fn drop(&mut self) {
        // Swap out our state with however we finished.
        let state_and_queue =
            self.state_and_queue.swap(self.set_state_on_drop_to, Ordering::AcqRel);

        // We should only ever see an old state which was RUNNING.
        assert_eq!(state_and_queue & STATE_MASK, RUNNING);

        // Walk the entire linked list of waiters and wake them up (in lifo
        // order, last to register is first to wake up).
        unsafe {
            // Right after setting `node.signaled = true` the other thread may
            // free `node` if there happens to be has a spurious wakeup.
            // So we have to take out the `thread` field and copy the pointer to
            // `next` first.
            let mut queue = (state_and_queue & !STATE_MASK) as *const Waiter;
            while !queue.is_null() {
                let next = (*queue).next;
                let thread = (*queue).thread.take().unwrap();
                (*queue).signaled.store(true, Ordering::Release);
                // ^- FIXME (maybe): This is another case of issue #55005
                // `store()` has a potentially dangling ref to `signaled`.
                queue = next;
                thread.unpark();
            }
        }
    }
}

WaiterQueue的析构方法会先设置once的状态，随后依次唤醒等待队列里的线程。
